1.2 Three Critical Findings

Finding 1 — computeLikelihoodBranchSIMD was hidden by -m GTR+G4. This function computes the total log-likelihood of the tree (no derivatives) and is called heavily during ModelFinder — a model selection step. With -m GTR+G4, ModelFinder is skipped, so this function showed 0.0 seconds. Finding 2 — OpenMP synchronisation is the #1 bottleneck in multi-threaded runs. 2. Background: What Is IQ-TREE?

IQ-TREE is a program used by biologists and bioinformaticians to build phylogenetic trees — diagrams that show the evolutionary relationships between species, genes, or other biological entities. Given a set of DNA or protein sequences from different species, IQ-TREE finds the tree that best explains how those sequences evolved.

2.1 Why Is It So Computationally Expensive?

At its core, IQ-TREE must answer: "How likely is it that this particular tree produced the DNA sequences we observe?" This question is answered by computing the phylogenetic likelihood — a number that scores how well a candidate tree explains the data.

The problem is that IQ-TREE must compute this likelihood thousands of times: once for every possible tree topology, once for every branch length adjustment, and once for every model parameter it tests. For a dataset with 50 species and 5,000 DNA positions, a single likelihood calculation involves:

  • 5,000 independent alignment patterns (each requiring ~60 floating-point operations)
  • 97 internal tree branches (each requiring its own likelihood calculation)
  • 3–5 Newton-Raphson iterations per branch (for optimising branch length)
  • ~88 model tests during ModelFinder (each requiring a full tree traversal)

The total adds up to billions of floating-point operations — and all of this happens inside five functions in a single file called phylokernelnew.h.

2.2 Key Terminology

Glossary for non-specialists:
  • SIMD / AVX — "Single Instruction, Multiple Data". A CPU technique that processes 4 numbers at once instead of one, like a cashier scanning 4 items simultaneously.
  • IPC — "Instructions Per Cycle". Measures how efficiently the CPU works. Higher is better (modern CPUs can do ~2–4 IPC).
  • Frontend stall — The CPU is waiting to fetch its next instruction from memory, like a factory worker waiting for the next part to arrive on the conveyor belt.
  • Backend stall — The CPU has the instruction but is waiting for data to arrive from memory. Like a worker with the blueprint but no materials.
  • OpenMP — A technology for splitting work across multiple CPU cores. #pragma omp parallel for tells the compiler "split this loop across all available threads".
  • GPU — A specialised processor with thousands of tiny cores, excellent for doing the same simple math on many data points simultaneously.
  • HIP / ROCm — AMD's programming framework for GPUs (equivalent to NVIDIA's CUDA).
  • ModelFinder — An IQ-TREE feature that tests ~88 DNA substitution models to find the best one. Skipped with -m GTR+G4.
  • NNI — "Nearest-Neighbor Interchange". A tree search move that swaps subtrees to find a better topology.
  • Dot product — A mathematical operation that multiplies two lists of numbers element-by-element and sums the results. Central to likelihood calculation.

2.3 How Profiling Works

Profiling is like putting a stopwatch on every part of a program. Two tools were used:

  • Intel VTune — Periodically samples which function the CPU is executing (every 10 milliseconds). Over thousands of samples, this reveals where time is spent. Used on hpc-01.
  • Linux perf — Reads CPU hardware counters (IPC, cache misses, branch mispredictions) at 997 samples per second. Provides both hotspot data and micro-architectural metrics. Used on both hpc-01 and Setonix.

4. Wall-Clock Scaling: 1T → 4T → 8T

Wall-clock time is the actual elapsed time — how long you wait for the result. When we add more threads, we expect the program to finish faster. But the speedup is never perfect because of overheads in coordinating threads.

4.1 Scaling Efficiency

Ideal scaling means that 4 threads should be 4× faster, and 8 threads should be 8× faster. Here we measure how close reality matches the ideal:

Why doesn't it scale linearly? The 8-thread run uses more than double the total CPU cycles of the 1-thread run (484.5s vs 230.8s). That extra 253.7 seconds of wasted CPU time comes from three sources: OpenMP barrier synchronisation (threads waiting for each other), cache coherence traffic (cores invalidating each other's cached data), and instruction-cache thrashing (the large code footprint doesn't fit in the I-cache when shared across many threads).

6. OpenMP Threading Bottleneck Analysis

6.1 The Problem: Thread Spin-Wait Dominates Multi-Threaded Runs

From the VTune Web UI top-down call stacks, the single most expensive entry in every multi-threaded run is libgomp.so.1!func@0x25820 — the OpenMP thread spin-wait function where worker threads busy-wait at barrier synchronisation points:

Call Stack (from VTune Web UI):

libgomp.so.1!func@0x25820        ← SPIN-WAIT (busy loop)
 libgomp.so.1!func@0x22640       ← thread pool manager
  libc.so.6!func@0x9c720         ← pthread internals
   libc.so.6!func@0x129c40       ← clone (thread creation)

This shows threads sitting in a busy-wait loop between OpenMP parallel regions. Think of it like a team of workers at a factory: they can all work on a task simultaneously, but between tasks they must wait at a meeting point until everyone is ready for the next task. The more frequently tasks change, the more time is wasted waiting.

6.3 Why the Overhead Is So High

IQ-TREE's parallelisation strategy creates frequent, small parallel regions. Each kernel function uses #pragma omp parallel for to split the pattern loop across threads. Between kernel calls (during serial Newton-Raphson control flow, tree traversal, etc.), worker threads spin waiting for the next parallel region:

NNI Search Loop (serial):
  for each branch (~97 branches):
    ┌─ computeLikelihoodBufferSIMD ── #pragma omp parallel for ──┐
    │  (pattern loop split across threads)                       │
    └──────────── barrier (threads wait) ────────────────────────┘
    for each Newton-Raphson iteration (3-5×):
       ┌─ computeLikelihoodDervSIMD ── #pragma omp parallel for ─┐
       │  (pattern loop split across threads)                     │
       └──────────── barrier (threads wait) ──────────────────────┘

With ~97 branches × ~4 Newton iterations = ~500 parallel region entries per NNI round. Each entry incurs thread wake-up, barrier synchronisation, and cache warm-up costs. For the medium_dna dataset (5,000 patterns ÷ 4 AVX width = 1,250 iterations), each parallel region processes only ~160 iterations per thread at 8T — not enough work to amortise the fork-join overhead.

8. The Five Hot Functions — Detailed Breakdown

All five functions live in a single file: phylokernelnew.h (3,688 lines). Together they consume >95% of total CPU time. Understanding what each function does is essential for designing GPU replacements.

8.1 computeLikelihoodDervSIMD — #1 Target (40–52% of CPU)

Purpose: Computes the log-likelihood, its first derivative (df), and second derivative (ddf) with respect to a single branch length. Called thousands of times during tree search — once per branch per Newton-Raphson optimisation step.

In plain English: Given the current tree and a specific branch, this function asks "How good is this branch length, and in which direction should we adjust it?" The first derivative tells us the direction, the second derivative tells us the step size.

Hot Loop (line ~2476)

// For each alignment pattern (e.g., 5,000 patterns for medium_dna):
for (ptn = ptn_lower; ptn < ptn_upper; ptn += VectorClass::size()) {
    theta = theta_all + ptn * block;

    // Compute 3 dot products simultaneously:
    //   lh_ptn  = Σ val0[i] × theta[i]   (likelihood)
    //   df_ptn  = Σ val1[i] × theta[i]   (1st derivative)
    //   ddf_ptn = Σ val2[i] × theta[i]   (2nd derivative)
    dotProductTriple(val0, val1, val2, theta, lh_ptn, df_ptn, ddf_ptn);

    // Normalise and accumulate
    lh_ptn   = 1.0 / (abs(lh_ptn) + ptn_invar[ptn]);
    df_frac  = df_ptn  * lh_ptn;
    ddf_frac = ddf_ptn * lh_ptn;
    my_df  += df_frac  * ptn_freq[ptn];
    my_ddf += (ddf_frac - df_frac * df_frac) * ptn_freq[ptn];
}

Operations per pattern (DNA, nstates=4, ncat=4): 3 × dot product of size 16 = 48 multiply-adds, plus 1 division, 6 multiplies, 3 additions = ~60 FLOPs per pattern.

GPU Suitability: ★★★★★ EXCELLENT Each pattern is completely independent — no pattern depends on any other pattern's result. Pure arithmetic with no branching. Regular memory access. The val0/val1/val2 arrays (16 doubles for DNA) are small enough to fit in GPU shared memory. 5,000 patterns map perfectly to 5,000 GPU threads.

8.2 computePartialLikelihoodSIMD — #2 Target (9.5–18.3%)

Purpose: Computes partial likelihood vectors at an internal tree node by combining information from its two child subtrees. Called once per internal node during tree traversal.

In plain English: For every branch point in the tree, this function combines the evidence from the two subtrees below it. It's like asking "Given what we know about the left subtree and the right subtree, what is the probability of each DNA base at this ancestral node?"

Three cases depending on node types:

  1. TIP-TIP (cherry): Two leaf children — cheapest case
  2. TIP-INTERNAL: One leaf + one internal child — intermediate
  3. INTERNAL-INTERNAL: Two internal children — most expensive case

Internal-Internal Hot Loop (line ~1855)

for (ptn = ptn_lower; ptn < ptn_upper; ptn += VectorClass::size()) {
    partial_lh_left  = left->partial_lh  + ptn * block;
    partial_lh_right = right->partial_lh + ptn * block;

    for (c = 0; c < ncat_mix; c++) {
        for (x = 0; x < nstates; x++) {
            // tmp[x] = (eleft · plh_left) × (eright · plh_right)
            dotProductDualVec(eleft, plh_left, eright, plh_right, tmp[x]);
        }
        // Transform back: partial_lh = inv_evec × tmp
        productVecMat(tmp, inv_evec, partial_lh);
    }
    // Numerical rescaling check
}

Operations per pattern: 4 rate categories × 4 dual dot products + 4 matrix-vector multiplies (4×4, 64 FMAs each) = ~480 FLOPs per pattern.

GPU Suitability: ★★★★☆ VERY GOOD Dense arithmetic, independent patterns, regular memory access. Caveat: tree traversal order creates data dependencies — a child node must be computed before its parent. GPU kernels must be launched per-node in post-order traversal.

8.3 computeLikelihoodBufferSIMD — #3 Target (3.8–8.2%)

Purpose: Computes the "theta buffer" — an element-wise product of partial likelihood vectors from two adjacent nodes. Computed once per branch and cached for reuse by computeLikelihoodDervSIMD.

In plain English: Before optimising a branch length, the program multiplies two lists of numbers together, element by element. The result is stored and reused 3–5 times during Newton-Raphson iterations.

Hot Loop (line ~2210)

for (ptn = ptn_lower; ptn < ptn_upper; ptn += VectorClass::size()) {
    theta = theta_all    + ptn * block;
    plh_n = node->partial_lh + ptn * block;
    plh_d = dad->partial_lh  + ptn * block;

    for (i = 0; i < block; i++)
        theta[i] = plh_n[i] * plh_d[i];  // element-wise product
}
GPU Suitability: ★★★★★ EXCELLENT Pure element-wise multiply. No branching, no reductions, no data dependencies. This is a "hello world" GPU kernel — trivial to implement.

8.4 computeLikelihoodBranchSIMD — #4 Target (0.3%)

Purpose: Computes the total log-likelihood of the tree. Unlike DervSIMD (which computes derivatives for Newton-Raphson), this computes only a single dot product per pattern. Called during ModelFinder when testing different substitution models.

9. Call Hierarchy: How Kernels Are Invoked

Understanding the call hierarchy shows when and why each kernel function is called during a tree search. This is critical for designing the GPU offload strategy — we need to know which functions are called together and how data flows between them.

doTreeSearch  [tree search iterations]
 └─ doNNISearch  [evaluate all NNI moves]
     └─ evaluateNNIs  [score each NNI candidate]
         └─ getBestNNIForBran  [per-branch NNI evaluation]
             └─ optimizeOneBranch  [Newton-Raphson on branch length]
                 └─ minimizeNewton  [Newton optimisation loop]
                     └─ computeFuncDerv
                         ├─ computeLikelihoodBufferSIMD (3.8%)
                         │   [Called ONCE per branch — computes theta cache]
                         └─ computeLikelihoodDervSIMD (40.1%)
                             [Called 3-5× per branch — Newton iterations]

computePartialLikelihoodSIMD (18.3%)
   [Pre-search, via computeTraversalInfo — tree traversal]

computeLikelihoodBranchSIMD (0.3%)
   [ModelFinder, via testModel → computeLikelihood]
Key insight for GPU design: computeLikelihoodBufferSIMD is called once and then computeLikelihoodDervSIMD is called 3–5 times reusing the same theta buffer. On a GPU, if we keep the theta buffer in GPU memory after Buffer computes it, the subsequent Derv calls avoid any CPU↔GPU data transfer. This is the concept of GPU-resident data.

11. GPU Offload Strategy

The profiling data makes a compelling case for GPU offloading. The five hot functions are embarrassingly parallel (each alignment pattern can be processed independently), and the CPU struggles with multi-threading due to OpenMP overhead, cache coherence, and instruction-cache pressure.

11.1 Priority List

PriorityFunctionDefault 1T TimeGPU SuitabilityNotes
#1computeLikelihoodDervSIMD 92.6s (40.1%) ★★★★★ 3 dot products/pattern, pure arithmetic
#2computePartialLikelihoodSIMD 42.2s (18.3%) ★★★★☆ Dense arithmetic; tree-order dependencies
#3computeLikelihoodBufferSIMD 8.7s (3.8%) ★★★★★ Trivial element-wise multiply
#4computeLikelihoodBranchSIMD 0.76s (0.3%) ★★★★★ 1 dot product + log reduction
#5computeLikelihoodFromBufferSIMD 0.41s (0.2%) ★★★★★ Wrapper for BranchSIMD with buffer

Combined kernel coverage: Priorities #1–#5 account for ~63% of VTune CPU time (direct attribution) and ~95% when including inlined AVX intrinsics.

11.2 Why GPU Beats Multi-Threading Here

The multi-threaded profiling data proves the CPU cannot scale efficiently on this workload:

ProblemEvidenceGPU Solution
OpenMP overhead 22–29% of CPU at 4T No fork-join; persistent kernel execution
Frontend stalls 52% (1T) → 70% (8T) Simple GPU kernels; no instruction-cache pressure
Backend stalls 17% (1T) → 48% (8T) No cache coherence (thread-private registers)
IPC degradation 1.63 (1T) → 0.89 (8T) Thousands of wavefronts hide latency via warp scheduling
Parallel efficiency 45% at 8T >90% with 5,000 patterns mapped to GPU threads

11.3 Kernel Designs

Priority 1 — computeLikelihoodDervSIMD

  • One GPU thread per alignment pattern
  • Each thread: dotProductTriple → normalise → accumulate
  • Warp/wavefront-level reduction for df/ddf sums (__shfl_down on AMD's wavefront-64)
  • val0/val1/val2 (16 doubles for DNA) in shared memory; theta_all in global HBM
  • Returns 2 doubles (df, ddf) to CPU

Priority 2 — computePartialLikelihoodSIMD

  • One thread per pattern per internal node
  • Inner work: dotProductDualVec + productVecMat per rate category
  • One kernel launch per node in post-order traversal
  • partial_lh arrays stay resident on GPU (avoid PCIe round-trips)

Priority 3 — computeLikelihoodBufferSIMD

  • Trivial element-wise multiply kernel
  • If partial_lh already on GPU (from Priority 2), zero PCIe transfer
  • Can be fused with the DervSIMD kernel launch

14.5 Key Findings from Setonix Profiling

Finding 4 — Trento eliminates the frontend stall bottleneck that crippled Sandy Bridge. On Sandy Bridge, frontend stalls consumed 52–70% of pipeline cycles (instruction-cache pressure from large templated AVX code). On Trento, frontend stalls are only 2–3% — a 30× reduction. The Zen 3 microarchitecture's larger instruction cache (32 KB L1i per core vs Sandy Bridge's shared 32 KB), deeper op-cache (4K entries), and better branch prediction handle the templated code far more efficiently.
Finding 5 — OpenMP overhead is proportionally higher on Setonix. Because Trento executes useful work so much faster (2.3 IPC vs 1.07), the relative cost of OpenMP synchronisation actually increases: 9.67% of perf samples on Setonix vs 6.44% on hpc-01. The absolute wall-clock time wasted is still lower (Setonix finishes the whole run in 26.9s), but the proportional overhead suggests that GPU offloading — which eliminates fork-join overhead entirely — will be even more beneficial on Trento.
Finding 6 — GPU offload targets are confirmed architecture-independent. The function hotspot ranking is identical on both platforms:
  1. computeLikelihoodDervSIMD — ~39% on both
  2. computePartialLikelihoodSIMD — ~33% on both
  3. computeLikelihoodBufferSIMD — ~10% on both
This means GPU kernels designed for one platform will benefit equally on the other.
