1. What Is IQ-TREE?

IQ-TREE is a program used by biologists to build evolutionary trees — diagrams showing how species are related to each other. Given DNA sequences from multiple species, IQ-TREE calculates which family tree best explains the observed differences between them.

This calculation — called the phylogenetic likelihood — is extremely expensive. IQ-TREE must evaluate it thousands of times while searching for the best tree. For a dataset with 50 species and 5,000 DNA positions, each evaluation involves millions of arithmetic operations. Virtually all of this work happens inside five mathematical functions in a single source file called phylokernelnew.h.

This report profiles where IQ-TREE spends its time, compares performance across two HPC systems, and explains why moving these calculations to a GPU should deliver significant speedups.

2. Where Time Is Spent

Profiling shows that more than 95% of total CPU time is consumed by just five functions. These functions perform the core mathematical calculations — dot products, matrix multiplications, and logarithms — that evaluate how well a candidate tree fits the DNA data.

Rank Function What It Does (Plain English) % of Time
#1 computeLikelihoodDervSIMD Calculates how much the tree score changes when a branch length is adjusted. Called thousands of times during the tree search. 40–52%
#2 computePartialLikelihoodSIMD Calculates intermediate probability values at each internal node of the tree, combining information from child nodes. 10–47%
#3 computeLikelihoodBufferSIMD Multiplies two arrays of numbers together, element by element. Prepares data for function #1. 4–13%
#4 computeLikelihoodBranchSIMD Calculates the overall tree score. Used when testing which DNA model fits best (ModelFinder phase). 0–0.4%
#5 computeLikelihoodFromBufferSIMD A wrapper that calls #4 using precomputed data from #3. 0–0.3%
Key takeaway: Functions #1 and #2 alone account for 60–90% of total runtime. Everything else in IQ-TREE — file I/O, tree rearrangement logic, model setup — is negligible by comparison.

The percentage ranges in the table above reflect differences between run configurations (e.g., whether ModelFinder is active) and thread counts. The ranking is consistent across both HPC systems.

  • IPC (Instructions Per Cycle) — how many instructions the CPU completes per clock tick. Higher is better. Maximum on Sandy Bridge is ~4.0.

4.3 Why Adding Threads Doesn't Help Much (hpc-01)

With 8 threads, hpc-01 achieves only 3.6× speedup instead of the ideal 8.0×. The profiler shows three reasons:

ProblemImpactExplanation (Plain English)
Thread waiting 22–29% of CPU time wasted Worker threads sit idle between tasks, waiting for the next batch of work. IQ-TREE creates ~500 small parallel work batches per search round, and the coordination overhead between batches is significant.
Instruction cache pressure Frontend stalls rise from 52% to 70% Multiple threads compete for the CPU's instruction cache, causing the CPU to frequently stall while fetching instructions from slower memory.
Data cache coherence Backend stalls rise from 17% to 48% When multiple cores access nearby data, the hardware must keep caches synchronised. This "snooping" traffic slows everyone down.
Thread synchronisation is the #1 bottleneck in multi-threaded runs. On hpc-01 at 4 threads, the OpenMP thread management library consumes 78.2 seconds of CPU time (22.6% of total) — more than any individual kernel function. At 8 threads it rises to 94.3 seconds. On Setonix at 4 threads, OpenMP overhead is 9.7% — proportionally higher because the faster CPU makes sync costs more visible.

6. The Case for GPU Acceleration

The profiling data reveals a fundamental mismatch between IQ-TREE's workload and CPU architecture. A GPU avoids every bottleneck identified in Section 4:

CPU ProblemEvidenceHow GPU Solves It
Thread waiting overhead 22–29% of CPU wasted at 4+ threads No thread barriers. GPU threads are always running; the hardware scheduler switches between them instantly.
Instruction cache thrashing Frontend stalls 52% → 70% as threads increase GPU kernels are simple and small. Each thread runs a compact loop that fits easily in cache.
Cache coherence traffic Backend stalls 17% → 48% at 8 threads GPU threads use private registers. No need for cross-core cache synchronisation.
Poor parallel efficiency Only 45% at 8 threads 5,000 alignment patterns map directly to thousands of independent GPU threads, achieving >90% utilisation.

6.1 Why These Functions Are Ideal for GPU

The five hot functions share characteristics that make them excellent GPU candidates:

  • Embarrassingly parallel: Each alignment pattern (5,000 in our test dataset) is processed independently. There is no communication between patterns.
  • Arithmetic-heavy: 60–480 floating-point operations per pattern, with minimal branching or data-dependent control flow.
  • Regular memory access: Data is accessed in sequential order, which GPUs handle efficiently through coalesced memory reads.
  • Small working set: The entire dataset (~30 MB for 50 taxa) fits comfortably in GPU memory (MI250X has 128 GB per chip).

6.2 GPU Target Prioritisation

PriorityFunctionGPU SuitabilityRationale
#1 DervSIMD ★★★★★ Highest time share (40–52%). Three independent dot products per pattern. Pure arithmetic.
#2 PartialLikelihoodSIMD ★★★★☆ Second highest (10–47%). Dense arithmetic but requires processing tree nodes in order (parent after children).
#3 BufferSIMD ★★★★★ Trivial element-wise multiply. If data is already on GPU from #2, this has zero transfer cost.
#4 BranchSIMD ★★★★★ Single dot product + logarithm. Same structure as #1 but simpler.
#5 FromBufferSIMD ★★★★★ Thin wrapper around #4.

8. Effect of ModelFinder on the Profile

IQ-TREE can run in two modes:

  • Default pipeline: First tests ~88 DNA models to find the best fit (ModelFinder), then searches for the best tree.
  • Fixed model (GTR+G4): Skips model testing and goes straight to tree search.

This matters for profiling because ModelFinder activates function #4 (BranchSIMD) and dramatically increases the time spent in function #2 (PartialLikelihoodSIMD):

ModelFinder causes a 167% jump in PartialLikelihoodSIMD time. Each of the ~88 model tests requires recomputing all partial likelihoods across the entire tree from scratch. This makes function #2 a more significant GPU target for default-pipeline workloads.

How to Read This Report

  • Wall-clock time: Actual elapsed time — how long you wait for the result.
  • CPU time: Total processor time across all cores. With 4 threads, CPU time ≈ 4× wall time (plus overhead).
  • IPC (Instructions Per Cycle): CPU efficiency metric. Higher = better. Theoretical max ~4.0.
  • Frontend stalls: CPU idle time caused by instruction fetch/decode delays. Lower = better.
  • Backend stalls: CPU idle time caused by memory access or cache synchronisation. Lower = better.
  • Speedup: How many times faster vs single-threaded. 4 threads with 2.5× speedup = 62.5% parallel efficiency.
  • perf: Linux profiling tool that reads hardware performance counters built into the CPU.
  • VTune: Intel's profiling tool that samples which function the CPU is running every 10 ms.
  • OpenMP: A library for splitting work across multiple CPU cores using #pragma directives.
  • HIP/ROCm: AMD's GPU programming framework (equivalent to NVIDIA's CUDA).
