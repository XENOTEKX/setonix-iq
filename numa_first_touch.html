IQ-TREE 3 — NUMA First-Touch (R1+R2) on Gadi vs Setonix

xlarge_mf.fa (200 taxa × 100 000 bp, GTR+G4) — Sapphire Rapids vs Zen3 Milan, full-node thread sweep
Run window: 2026-05-08 · Author: Ayush Samuel · iqtree3 build-profiling-clang (icx) on Gadi, AOCC 5.1.0 on Setonix

1. The problem and the fix

Linux uses first-touch NUMA placement: the first thread to write a virtual page decides which NUMA node holds the physical page for the rest of the process's lifetime. IQ-TREE was filling several large per-pattern arrays from a single thread (the master), pinning every page to one NUMA node. When the parallel kernel later spawned threads on other sockets/CCDs, every read from those arrays became a remote-DRAM load with cross-socket coherence cost.

Eight source edits across two files convert the kernel's OpenMP scheduling and one-time/per-call buffer fills to a single schedule(static) idiom, so each Linux page is first-touched by the worker thread that will own it for the rest of the run. Same edits, same line ranges, same comments on Gadi and Setonix.

Why it works. R1 fixes the one-time init arrays (ptn_freq, ptn_invar) so each thread first-touches its own pattern range. R2a does the same for the per-call _pattern_lh_cat zero-fill that runs on every NNI iteration. R2b stops the dynamic scheduler from re-shuffling packets between threads on every kernel call, which was defeating any locality R1 established. Together they enforce a single (thread → static pattern range) mapping on both write and read sides of every page.

1.2 R1a — tree/phylotreesse.cpp:546 · computePtnFreq

Read every iteration of the likelihood Newton-Raphson loop (#1 hot func, ~40% CPU). Serial master-thread fill pins all ptn_freq pages on socket 0; fix: parallel-static fill so each worker first-touches the pages it will own for the rest of the run.

Before 
void PhyloTree::computePtnFreq() {
    if (ptn_freq_computed) return;
    size_t nptn = aln->size();
    for (size_t ptn = 0; ptn < nptn; ptn++)
        ptn_freq[ptn] = (*aln)[ptn].frequency;
    ptn_freq_computed = true;
}
After 
void PhyloTree::computePtnFreq() {
    if (ptn_freq_computed) return;
    size_t nptn = aln->size();
#ifdef _OPENMP
#pragma omp parallel for schedule(static)
#endif
    for (size_t ptn = 0; ptn < nptn; ptn++)
        ptn_freq[ptn] = (*aln)[ptn].frequency;
    ptn_freq_computed = true;
}

1.3 R1b — tree/phylotreesse.cpp:577 · computePtnInvar

Read every iteration of computeLikelihoodDervSIMD and computeLikelihoodBranchSIMD. The original memset is replaced with a parallel-static zero-fill (first-touch), and the final scale loop is also parallelised so the scaling writes match the same thread-to-range mapping.

1.4 R2a — tree/phylotreesse.cpp:1294-1295 · computeLikelihoodBranchEigen

Per-call _pattern_lh_cat zero-fill. Buffer size: nptn × ncat_mix × sizeof(double) ≈ 1.6 MB for xlarge_mf. Called once per computePatternLhCat invocation, which happens on every NNI candidate evaluation — thousands of times per tree-search iteration. The downstream reduction loops at lines 1321 and 1365 already use schedule(static); this patch makes the upstream zero-fill match the same thread-to-range mapping so those pages are local when the reductions run.

Before 
double prob_const = 0.0;
memset(_pattern_lh_cat, 0, sizeof(double) * nptn * ncat_mix);
// ... downstream reduction loops at :1321 and :1365 use schedule(static) ...
After 
double prob_const = 0.0;
// NUMA first-touch: parallel-static zero-fill so _pattern_lh_cat pages are
// placed on the same NUMA node as the worker that later reads/writes them
// in the static-scheduled reduction loops below (lines 1321 and 1365).
{
    size_t lh_cat_n = nptn * ncat_mix;
#ifdef _OPENMP
#pragma omp parallel for schedule(static)
#endif
    for (size_t k = 0; k < lh_cat_n; k++)
        _pattern_lh_cat[k] = 0.0;
}

1.5 R2b — tree/phylokernelnew.h ×5 · kernel packet loops

One-line schedule change, applied at five separate sites. The num_packets loop is sized by computeBounds to be roughly equal-cost per packet, so the load-balancing rationale that originally motivated schedule(dynamic,1) is weak in practice. Any residual imbalance loss is dwarfed by the NUMA-locality win at every thread count tested.

Before — all 5 sites 
#pragma omp parallel for schedule(dynamic,1) num_threads(num_threads)
for (int packet_id = 0; packet_id < num_packets; packet_id++) {
    // ... per-packet kernel work ...
}
After — all 5 sites 
#pragma omp parallel for schedule(static) num_threads(num_threads)
for (int packet_id = 0; packet_id < num_packets; packet_id++) {
    // ... per-packet kernel work ...
}

5. Topology and the 32T regression

The only thread count where R2 slowed Gadi down is 32T (+8.0%). On Sapphire Rapids 32 threads fit entirely inside socket 0 — there is no cross-socket NUMA pressure for the patch to fix, and the strict schedule(static) partition is slightly worse for load balancing than the original schedule(dynamic,1) when all threads share local DRAM.

On Setonix the NUMA boundary appears every 8 cores (CCD), so even at 16T the dynamic scheduler was already paying a remote-DRAM cost. R2 therefore helps at every thread count tested on Setonix (−34.7% at 32T, −56.4% at 64T, −68.9% at 128T).

The cross-socket cliff on Gadi sits at ~52T. Below that, R2 is neutral-to-negative; above it, the benefit compounds. This is a property of the chip topology, not of the patch — the patch is correct on both platforms; the workload just doesn't need it below the first NUMA boundary.
