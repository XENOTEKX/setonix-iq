# GitHub Actions CI/CD Overhaul — Implementation Plan

## Problem

Currently, `serve.py` generates `dashboard.html` and `docs/index.html` on the Setonix login node, then those generated files are committed and pushed alongside the raw data. This is fragile:

1. Generated HTML files cause merge conflicts
2. The Setonix login node must have all dependencies to generate the dashboard
3. No separation between data (source of truth) and presentation (generated)

## Solution

**Data-only pushes from Setonix → GitHub Action auto-generates the website.**

### What gets pushed from Setonix
- `logs/runs/*.json` — run data (timing, metrics, hotspots, flamegraphs)
- `logs/profiles/*.json` — deep profile data (counters, stacks)
- `serve.py` — dashboard generator (source)
- `website/index.html` — HTML template (source)
- `website/api/data.py` — data parsing layer (source)

### What gets generated by GitHub Action (NOT committed from Setonix)
- `docs/index.html` — GitHub Pages deployment
- `dashboard.html` — convenience copy
- `website/api/runs.json` — reference JSON

## Changes

### 1. `.gitignore` updates
Add these entries to prevent generated files from being tracked:
```
# Generated by GitHub Action (do not commit from Setonix)
dashboard.html
docs/
PROFILING_REPORT.html
```

Remove these files from git tracking:
```bash
git rm --cached dashboard.html docs/index.html PROFILING_REPORT.html
```

### 2. GitHub Action: `.github/workflows/build-dashboard.yml`
**Trigger:** Push to `main` when any of these change:
- `logs/**` (new run/profile data)
- `serve.py` (generator code)
- `website/**` (template or data layer)

**Steps:**
1. Checkout repository
2. Setup Python 3.x (no external deps — serve.py uses only stdlib)
3. Run `python3 serve.py` (reads `logs/runs/*.json` via data.py fallback path)
4. Deploy `docs/` to GitHub Pages using `actions/deploy-pages`

**Error handling:**
- Validate serve.py exit code
- Check that `docs/index.html` was actually generated
- Validate generated HTML is non-empty and contains expected markers
- Post error annotation on failure

### 3. `serve.py` — no changes needed
The existing fallback path in `data.py` already handles the CI case:
- When `website/results/` doesn't exist → reads `logs/runs/*.json`
- When `website/deep_profiles/` doesn't exist → reads `logs/profiles/*.json`
- This is exactly what happens in GitHub Actions (no Setonix symlinks)

### 4. `start.sh` — update `sync_to_github()`
Remove the `python3 serve.py` call from `cmd_start()` before the push.
The push now just sends data; the Action generates the website.
Keep `serve.py` callable for local preview (`./start.sh generate` still works).

### 5. GitHub Pages configuration
- Source: GitHub Actions (not "Deploy from branch")
- The workflow uses `actions/upload-pages-artifact` + `actions/deploy-pages`

## Testing
1. Run `python3 serve.py` locally to verify it works with only `logs/` data
2. Push to GitHub and verify the Action runs successfully
3. Check https://xenotekx.github.io/setonix-iq/ loads correctly

## Rollback
If the Action fails, `docs/index.html` can still be generated locally and pushed manually by temporarily removing it from `.gitignore`.
