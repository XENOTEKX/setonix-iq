# CI/CD Pipeline — Implementation Plan

## Architecture

```
Setonix (Pawsey HPC)          GitHub                    hpc-01 (ANU HPC)
─────────────────────     ─────────────────────     ─────────────────────
Run profiling job    ──►  Receive push             
./start.sh sync      ──►  Trigger Action           
  (data-only push)        ├─ python3 serve.py      
                          ├─ Validate output       
                          └─ Commit dashboard  ──►  Cron pulls every 5min
                              [skip ci]             ./host.sh refresh
                                                    Serve on HTTP :8080
```

**Key constraint:** Setonix blocks incoming TCP, SSH forwarding, and unapproved servers (Pawsey network policy). GitHub Actions can't SSH into hpc-01 (ANU network). Solution: hpc-01 pulls from GitHub on a cron schedule.

## What gets pushed from Setonix
- `logs/runs/*.json` — run data (timing, metrics, hotspots, callstacks)
- `logs/profiles/*.json` — deep profile data (counters, stacks)
- `serve.py`, `website/index.html`, `website/api/data.py` — source code

## What gets generated by GitHub Action
- `dashboard.html` — self-contained dashboard (committed back to repo)
- `docs/index.html` — same dashboard in docs/ directory
- `website/api/runs.json` — reference JSON

## Components

### 1. `.github/workflows/build-dashboard.yml`
**Trigger:** Push to `main` when `logs/`, `serve.py`, or `website/` change.

**Steps:**
1. Checkout repository
2. Setup Python 3.12
3. Run `python3 serve.py` (reads `logs/runs/*.json` via data.py fallback)
4. Validate output (file exists, size > 1KB, contains expected marker)
5. Commit generated files back with `[skip ci]` to prevent loop

**Error handling:**
- Validate serve.py exit code
- Check `docs/index.html` was generated and is non-empty
- Post `::error::` annotation on failure

### 2. `start.sh` (Setonix)
Pushes data only. Does NOT call `serve.py` before push.
`./start.sh generate` available for local preview.

### 3. `host.sh` (hpc-01)
- `./host.sh start` — Start HTTP server (screen + python3 -m http.server)
- `./host.sh cron` — Install 5-min cron job to pull + regenerate
- `./host.sh refresh` — Manual pull + regenerate
- `./host.sh stop` — Stop server

### 4. `serve.py` — no changes needed
The `data.py` fallback path handles the CI case:
- No `website/results/` → reads `logs/runs/*.json`
- No `website/deep_profiles/` → reads `logs/profiles/*.json`

## One-time setup

### hpc-01 (run once):
```bash
git clone https://github.com/XENOTEKX/setonix-iq.git ~/setonix-iq
cd ~/setonix-iq
./host.sh start    # starts HTTP server on port 8080
./host.sh cron     # installs 5-min auto-refresh
```

### GitHub repo settings:
No Pages config needed (private repo, commit-back approach).

## Testing
1. Push data from Setonix: `./start.sh sync`
2. Check GitHub Actions tab — Action should generate + commit dashboard
3. On hpc-01: `./host.sh refresh` or wait for cron
4. Open `http://hpc-01:8080` in browser

## Rollback
Generate locally on Setonix: `./start.sh generate` then `git add -f dashboard.html && git commit && git push`
